## 2026-04-15 — Быстрый аудит багов и оптимизаций

### Найденные баги

1. **Daily Challenge считает поражение от бота как обычную победу/финал матча**
   - В `placeNewMark()` при любом победителе вызывается `triggerWin(winner)`.
   - В `triggerWin()` для daily всегда используется `stopDailyTimer('win')` и win-оверлей, даже если победил `wraith` (бот).
   - Итог: в daily нет отдельного пути для сценария «бот победил», что ломает семантику режима-испытания.

2. **Переключение режима в setup не синхронизирует видимость ряда сложности бота**
   - Видимость `#setup-bot-difficulty-row` меняется в `syncAvailableModes()`.
   - Но `selectMode()` вызывает только `syncRankedLockState()` и не вызывает `syncAvailableModes()`.
   - Итог: при переключении, например, Duel -> Daily строка сложности может визуально остаться до следующего триггера синхронизации.

3. **Закрытие окна daily-инфо принудительно сбрасывает режим на Duel**
   - `closeDailyInfo()` всегда делает `selectMode('duel')`.
   - Итог: пользователь теряет выбранный режим без явного подтверждения, что создает лишнее трение в setup-флоу.

4. **`startDailyChallenge()` не проверяет, что dailyChallenge инициализирован**
   - Функция напрямую читает `dc.preBoard` после `const dc = dailyChallenge;`.
   - При нестандартном вызове (горячая клавиша/инъекция/неожиданное состояние UI) возможна ошибка времени выполнения.

### Возможные оптимизации

1. **Снизить лишние DOM-операции в `render()`**
   - Сейчас на каждом рендере удаляются и заново создаются предупреждающие рамки (`warnDivs`) и пересчитываются части UI.
   - Можно кэшировать состояние и обновлять только изменившиеся узлы.

2. **Упростить выбор доступных/активных mode-кнопок**
   - Логика активации кнопок режима в `syncAvailableModes()` и `selectMode()` частично дублируется.
   - Можно объединить в один источник правды для уменьшения риска рассинхронизации.

3. **Уменьшить количество полных проходов по полю в бот-логике**
   - `chooseBestBotMove()` и вспомогательные функции несколько раз строят массив пустых клеток/симулируют ходы.
   - Для 3x3 это недорого, но можно сократить аллокации и проходы для более чистого исполнения.

4. **Осторожнее с `innerHTML` для локализации статических текстов**
   - В `setLang()` используется `el.innerHTML = el.dataset[l]` для всех `[data-ru]`.
   - При расширении проекта безопаснее перейти на `textContent` там, где HTML-разметка не нужна.
